\section{Changes in Response to Feedback}

This section summarizes the changes made across project deliverables in
response to feedback from TAs, peer reviewers, and internal team review.
Each item identifies the feedback received, its source, and the specific
Revision 1 changes made in response. GitHub issue and pull request links
are included to support traceability to the corresponding tracked changes.

Peer review feedback was also considered during revision. When peer
feedback aligned with TA feedback or identified clear issues in clarity,
structure, traceability, or testability, the suggested changes were
incorporated. When peer feedback was ambiguous, conflicted with the
document's purpose, or introduced unnecessary implementation detail,
the team reviewed it but did not apply it. These decisions were made to
preserve consistency with project scope, black-box requirements
principles, and previously validated design choices.

\section{Design Iteration (LO11 (PrototypeIterate))}

Our first working prototype (MVP) used a simple pipeline: capture a screenshot,
run OCR, append extracted text to the user request, and send the combined prompt
to an LLM for a one-shot response. This validated feasibility, but testing
revealed recurring issues: OCR noise reduced reliability, one-shot responses were
hard to recover when a step failed, and behavior varied across interfaces.

From a design perspective, the main lesson from this stage was that the system
could demonstrate capability, but not yet dependable usability. The usability
study confirmed this broader lesson later: participants saw the value of the idea
quickly, but they still needed clearer entry points, better examples, and a more
consistent interaction model before the system felt easy to adopt.

\textbf{Iteration 1: one-shot flow to agent loop.} We introduced an agent loop
(reason $\rightarrow$ decide $\rightarrow$ act $\rightarrow$ observe
$\rightarrow$ reflect) so the system could make progress in smaller steps,
inspect tool outputs, and recover from partial failures. We made this change to
reduce user burden, especially for users facing accessibility barriers who should
not need to repeatedly restate intent after one failed action.

This change also aligned with the usability findings that users often knew the
outcome they wanted, but were uncertain about the exact phrasing or the best
follow-up command. The agent loop gave the assistant more room to clarify,
recover, and continue the task instead of forcing the user to restart the
conversation.

\textbf{Iteration 2: tightly coupled integrations to MCP integrations.} As the
number of capabilities grew, integration code became difficult to scale and
maintain. We moved to MCP-backed integrations to support modular growth and
clearer boundaries between orchestration and external tools.

In the later version of this design, we also added prompt caching and MCP tool
search so the system could reuse useful context instead of repeatedly sending
large prompts. That reduced token usage, lowered operational cost, and made the
assistant more efficient when handling repeated or similar requests.

This decision was reinforced by the usability testing results because the system
needed to support realistic workflows like email, calendar, notes, and weather
without making each feature feel like a separate product. Modular integrations
allowed us to add and refine capabilities while preserving the same user-facing
interaction style.

\textbf{Iteration 3: separate interface logic to shared gateway.} Early versions
already used a shared backend or bridge, but the control paths were still not
general enough for all sources of work. We introduced a gateway architecture to
formalize many-to-one-many routing so that cron jobs, webhooks, GUI, TUI,
Discord relay, and future chat integrations could all enter the same runtime and
receive consistent results on the way out. This added a clean control layer on
top of the existing shared backend.

The usability study made this especially important because participants valued
the assistant most when it could support real tasks in a consistent way. If one
entry path behaved differently from another, the system would feel less
predictable and harder to learn. The gateway was therefore not only an
architecture improvement, but also a usability decision.

\textbf{Iteration 4: selected MCP flows to direct CLI tools.} For common,
high-frequency tasks, we converted some MCP interactions to local CLI-backed
tools. This reduced overhead and improved operational reliability for frequently
used workflows.

We also used tool search for MCPs in the final version to avoid loading or
describing unnecessary tools in every turn. This kept the active tool surface
smaller, which lowered token usage and reduced cost while still preserving access
to the right integrations when they were needed.

This change was influenced by the need to keep common actions responsive and
reliable. In testing, the most useful experiences were the ones where the user
could complete a task quickly without being interrupted by setup friction or
unclear transitions. Moving some high-value actions to CLI-backed tools helped
reduce that friction.

Overall, the design evolved from a proof-of-concept into a layered architecture
with explicit orchestration, modular integrations, and a shared runtime across
interfaces. Each iteration was a response to observed limitations and to the
same client objective: dependable, low-friction interaction for users who may
face barriers with traditional computer interfaces. The usability study evidence
(for example, command discoverability, follow-up ambiguity, and feature
discoverability findings) also directly influenced our iteration priorities and
documentation improvements, while the later prompt-caching and tool-search work
reduced token usage and made the system cheaper to operate.

In particular, the exploratory free-use sessions were useful because they showed
what users attempted when they were not constrained by scripted tasks. That
revealed where discoverability broke down and where supported features needed to
be surfaced earlier. Those observations pushed us to improve documentation, the
user manual, and the visibility of interface options in the final version.

\section{Design Decisions (LO12)}

\subsection{Limitations and Their Influence}

\begin{itemize}
	\item \textbf{OCR quality limitations:} Screenshot text extraction is not
	always accurate or complete. Because the usability study showed that users were
	already sensitive to unclear or unreliable responses, we prioritized iterative
	reasoning and verification instead of one-shot execution.
	\item \textbf{External dependency limitations:} Model providers and external
	integrations can fail, throttle, or change. This pushed us to modularize
	integrations and design for graceful degradation, which was important because
	participants valued completing real workflows without having to restart from
	scratch when something went wrong.
	\item \textbf{Multi-client limitations:} GUI, TUI, and Discord differ in
	interaction style and transport. This drove the decision to centralize runtime
	behavior in a gateway, since the usability work showed that users expected the
	same task to behave consistently regardless of the entry path they used, and
	we also needed one place to absorb cron jobs, webhooks, and future chat inputs.
\end{itemize}

\subsection{Assumptions and Their Influence}

\begin{itemize}
	\item \textbf{A1:} Natural-language interaction lowers the barrier for our
	target users compared to command memorization. This supported investing in
	agent-based orchestration, especially because usability testing showed that
	participants understood the value of the system quickly but struggled with how
	to phrase requests and follow-up commands.
	\item \textbf{A2:} Users may switch interaction surfaces while expecting
	equivalent behavior. This justified a shared backend rather than per-interface
	business logic, because the study indicated that users cared more about task
	completion than about which front end they were using.
	\item \textbf{A3:} The integration surface will continue to expand over time.
	This justified MCP modularity, with selective CLI conversion for high-value
	paths, so we could keep common workflows dependable while adding features.
\end{itemize}

\subsection{Constraints and Their Influence}

\begin{itemize}
	\item \textbf{Capstone time constraint:} We prioritized reliability,
	consistency, and core accessibility value over lower-priority feature breadth,
	because the usability study identified clarity and discoverability as the main
	friction points rather than missing advanced capabilities.
	\item \textbf{Team-size constraint:} Duplicated orchestration across clients was
	not sustainable, so we centralized execution paths.
	\item \textbf{Resource/cost constraint:} External API usage and operational
	overhead influenced our choice to keep some frequent operations local via CLI.
\end{itemize}

\subsection{Traceability from PoC to Rev 0 to Final Presentation}

The design evolution is explicitly traced across three development stages below.

\begin{enumerate}
	\item \textbf{PoC stage (initial prototype):}
	\begin{itemize}
		\item The initial prototype focused on direct computer control through MCP
		tools.
		\item Context management was limited: the system did not yet perform robust
		selection or structuring of relevant context, and large amounts of context
		were frequently compressed into a single prompt.
		\item Orchestration support for iterative recovery was minimal.
	\end{itemize}

	\item \textbf{Rev 0 stage:}
	\begin{itemize}
		\item A React GUI was introduced to improve usability and accessibility for
		non-technical users.
		\item Initial orchestration capabilities were added to reduce dependence on
		purely one-shot behavior and improve action sequencing.
		\item MCP integrations were retained and reorganized around concrete user
		workflows exposed through the GUI.
	\end{itemize}

	\item \textbf{Final presentation stage:}
	\begin{itemize}
		\item A shared gateway architecture was introduced as the central runtime for
		orchestration, routing, and session management.
		\item Interaction surfaces were expanded to \textbf{TUI, GUI, and Discord relay},
		all connected to the same backend behavior.
		\item This change reduced interface drift and improved behavioral
		consistency, maintainability, and reliability under multi-client operation.
	\end{itemize}
\end{enumerate}

\subsection{Challenge Level and Extras}

\textbf{Challenge level:} High. The project required coordinating asynchronous
agent behavior, multiple interfaces, and integration reliability while preserving
an accessibility-first product goal. The usability study reinforced that this was
not just a feature problem: users needed clearer entry points, clearer prompts,
and consistent task behavior across interfaces.

\textbf{Extras:} Usability testing and the user manual were completed extras
that directly influenced design priorities, especially consistency, recoverability,
and clarity for end users.

\section{Economic Considerations (LO23)}

There is a growing market for AI-assisted productivity and accessibility
tools. Adoption of AI-powered systems has increased significantly in
recent years as users seek more efficient and intuitive ways to interact
with software systems~\citep{ai_adoption}. In particular, tools that
reduce reliance on traditional interfaces and support natural language
interaction are becoming increasingly relevant.

Proxi is currently developed as an open-source system with no immediate
plans for commercialization. However, a potential commercial version
could target professionals, accessibility-focused users, and enterprise
environments that benefit from automation and simplified workflows.

If commercialized, estimated development and maintenance costs would
primarily include infrastructure for AI services, system integration,
and ongoing updates. A reasonable estimate for transitioning to a
production-ready system would be approximately \$20,000--\$40,000 CAD,
depending on hosting requirements, API usage, and long-term support.

A possible pricing model could involve a subscription-based approach,
with pricing in the range of \$10--\$25 per user per month. This aligns
with existing subscription-based productivity tools, which commonly fall
within a similar pricing range~\citep{pricing}.

Assuming an average price of \$15 per month, the system would require
approximately 150--300 active users over one year to recover initial
costs. Beyond this point, additional users would contribute to profit,
making the model scalable.

Since Proxi is currently open-source, the primary focus is on attracting
users rather than generating revenue. User adoption can be increased by:

\begin{itemize}
\item Providing clear documentation and onboarding through the user manual.
\item Demonstrating system capabilities through example workflows.
\item Leveraging usability testing results to improve user experience.
\item Sharing the project through GitHub and academic networks.
\end{itemize}

The potential user base is large. Globally, a significant portion of the
population benefits from assistive or accessibility-focused technologies,
and demand for accessible computing solutions continues to grow
~\citep{accessibility}. In addition, students and professionals seeking
more efficient workflows represent a broad secondary user group.

Overall, while Proxi is not currently positioned as a commercial product,
the system demonstrates strong potential for future adoption and
scalability. These estimates are preliminary and intended to demonstrate
feasibility rather than provide exact financial projections.

\section{Reflection on Project Management (LO24)}

This section reflects on how we managed schedule, scope, team
coordination, and verification throughout the capstone project. As
shown in the Development Plan and V\&V documents, our process relied on
both project-management tools and engineering tools: Git and GitHub for
version control, issue tracking, pull-request review, and traceability;
Discord for day-to-day coordination; and technical tools such as
Python, TypeScript/React, Playwright, OpenAI-based APIs, and linting
and testing workflows to support actual implementation and quality
control.

\subsection{How Does Your Project Management Compare to Your Development Plan}

Overall, our project management stayed reasonably close to what we
committed to in the Development Plan. We used the planned weekly team
meetings, rotating chair and note-taker roles, Discord for day-to-day
communication, and GitHub Issues, pull requests, and the project board
as the main tools for assigning work, reviewing changes, and keeping
progress visible. On the engineering side, the project also stayed well
aligned with the documented technology plan: Python for orchestration,
TypeScript/React for interface work, OpenAI-based language-model
support, Playwright and platform automation tools for task execution,
and Git-based review practices for quality control.

At the same time, the real project required more flexibility than the
initial plan suggested. The role table gave us a useful starting point,
but once implementation, testing, and documentation began to interact
more closely, people had to contribute across boundaries instead of
remaining in a single narrow role. Scope also had to be managed more
carefully over time. Rather than trying to pursue an overly broad vision
of a fully autonomous assistant, we focused the project around reliable
voice and text interaction, safe task execution, and a proof of concept
that was feasible within capstone constraints.

\subsection{What Went Well?}

One of the biggest strengths of our project management was that we used
simple tools consistently and for clear purposes. GitHub Issues, pull
requests, and the GitHub project board gave us a shared source of truth
for tasks, review status, and progress, while Discord helped us resolve
blockers quickly between meetings. On the engineering side, having a
clear tool stack also helped: Python and its supporting libraries made
rapid backend iteration possible, TypeScript/React supported a more
structured frontend, and tools such as Playwright and platform-specific
automation libraries let us test real computer interactions rather than
only describe them abstractly.

Another process that worked well was connecting project management to
verification rather than treating testing as a final-step activity. As
reflected in the V\&V Plan and V\&V Report, we used checklist-based
inspections, traceability matrices, and unit, integration, and system
tests as concrete management tools for deciding whether work was ready.
The coding-standard tools from the Development Plan, such as pylint,
Prettier, and ESLint, also supported consistency across the Python and
TypeScript codebases. Together, these tools improved accountability,
kept quality visible, and made the final stages of the project more
manageable because evidence and traceability were collected throughout
the term instead of being invented at the end.

\subsection{What Went Wrong?}

The main difficulty was that some parts of the project needed more
coordination effort than the original plan implied. Integrating AI
services, desktop and browser automation, frontend work, verification,
and documentation created more dependency between tasks than a simple
role table can capture. In practice, that meant the engineering tools
that made Proxi possible also added coordination overhead: external LLM
services can be variable, browser and OS automation can be brittle, and
a multi-language codebase requires more care when different components
must evolve together. As a result, some work could not progress as
cleanly in parallel as we first expected, and late-stage consistency
checking across documents took a significant amount of time and focus.

We also learned that scope control had to be more active than we had
initially assumed. Early ideas around Proxi could easily expand into a
much broader autonomous-assistant vision, but the team had to narrow the
focus toward the most important and demonstrably feasible workflows. In
addition, some workflow details, such as stronger branch conventions,
clearer traceability ownership, fuller tool justification, and earlier
use of process checklists, were only strengthened after external
feedback rather than being fully mature from the beginning.

\subsection{What Would you Do Differently Next Time?}

For a similar project, we would formalize several management practices
much earlier. We would define branch naming and review expectations from
the start, schedule more frequent internal milestone reviews before each
official deliverable, and assign an explicit owner for cross-document
traceability so the final reporting effort is distributed more evenly
across the term. We would also make even better use of management tools
such as GitHub issue templates, review checklists, and clearer status
tracking so process discipline does not depend only on memory or last-
minute cleanup.

We would also revisit feasibility risk and fallback scope at regular
checkpoints instead of only when deadlines become close. For a project
that depends on AI behaviour, external services, and automation tools,
it is especially important to decide early which workflows are
release-critical and which are optional extensions. Doing that sooner,
along with earlier automation of linting and testing expectations, would
improve time management, reduce last-minute scope pressure, and make
resource decisions more deliberate and effective.

\section{Ethics and Equity (LO18)}

Our team considered ethics and equity throughout the design and
development of Proxi, with a focus on accessibility, inclusivity, and
responsible use of AI-assisted technologies.

A key principle guiding the project was universal design. Proxi enables
users to interact with computer systems using natural language rather
than requiring technical knowledge of commands or interfaces. This
reduces barriers for users who may have limited technical experience,
including older adults and users with accessibility needs.

The system was designed to support equitable access by prioritizing
simplicity and clarity in interactions. For example, users are not
required to remember specific commands, and feedback is presented in a
way that is easy to understand. These design choices help ensure that
the system is usable by a broader range of users, rather than only those
with advanced technical skills.

Empathy played an important role in understanding stakeholder needs.
Personas representing accessibility-focused users and non-technical
users were used to guide requirements and design decisions. This helped
the team consider how different users might experience frustration,
confusion, or barriers when interacting with traditional systems, and
influenced the focus on intuitive interaction and error tolerance.

The team also considered ethical implications of using external AI
services. There is a potential risk that user data may be transmitted to
external systems. To address this, the system is designed to minimize
data sharing and ensure that only task-relevant information is used.
Users are also expected to be informed when external processing may
occur, supporting transparency and informed consent.

Despite these efforts, there are still limitations. The system relies on
off-the-shelf AI models, which may not perform equally well across all
users or contexts. This could lead to inconsistent experiences,
particularly for users with different speech patterns or accents. While
this limitation was recognized, it was not fully addressed within the
scope of the project.

Overall, the team made a conscious effort to design Proxi in a way that
promotes equitable access and usability, while also acknowledging areas
where further improvements are needed to better support all users.

\section{Reflection on Capstone}

This capstone was a valuable experience because it brought together many
parts of software engineering in a way that regular course work usually
keeps separate. It required us to move from isolated assignments to a
single evolving product shaped by real constraints, user needs,
iteration, verification, and team coordination. In that sense, the
project was not just about building Proxi, but also about learning how
to make technical decisions, respond to feedback, and refine a system
over time.

At the same time, several courses gave us an important foundation for
this work. They helped us approach requirements, architecture, testing,
accessibility, and security more systematically, even though the
capstone still required us to extend that knowledge in more practical
and open-ended ways.

\subsection{Which Courses Were Relevant}

Several courses were especially relevant to Proxi. SFWRENG 3RA3
(Software Requirements \& Security Considerations) was directly useful
for writing the SRS, thinking about stakeholder needs, and handling
privacy and safety concerns in the hazard analysis and traceability
work. SFWRENG 3S03 (Software Testing) was also highly relevant because
our project required structured verification through unit, integration,
system, and usability-related testing rather than relying on informal
checks.

Our software design courses, especially SFWRENG 3BB4 and SFWRENG 3A04,
were important for the architectural side of the project. Proxi depends
on modular decomposition, clear interface boundaries, and separation of
responsibilities across the gateway, integrations, memory, and user
interfaces. SFWRENG 4HC3 (Human-Computer Interfaces) was also very
relevant because accessibility and ease of use were central goals of
this project, and that background helped us think more carefully about
user workflows, feedback clarity, and the usability testing extra.
Finally, SFWRENG 4C03 (Computer Networks \& Security) reinforced the
importance of secure integrations, external services, and dependable
communication between system components.

\subsection{Knowledge/Skills Outside of Courses}

At the same time, some of the most important skills for this project
were learned outside formal coursework. We had to learn how to work
with modern LLM APIs, prompt design, and tool-calling orchestration in a
way that was reliable enough for a real assistant rather than a small
classroom prototype. We also had to gain hands-on experience with
browser and desktop automation tools and understand the practical
tradeoffs that come with external services, such as latency, token cost,
and service reliability.

Another major area of learning was practical accessibility and usability
evaluation. Coursework helped us think about interface design, but
running a real usability study, interpreting participant feedback, and
turning that feedback into documentation and workflow improvements
required more applied judgment than a typical course assignment. This
combination of course foundations and self-directed learning was one of
the most valuable parts of the capstone experience.
